# AjaxChatRoom 任务清单

## 背景

当前项目在同一浏览器中登录多个账号进行测试时，会话被同一个 cookie 覆盖，导致后端只识别最后一个登录的账号。要支持同一浏览器多个账号并行测试，需要改造会话管理和测试方式。

## 目标

- 支持同一浏览器多个账号并行测试
- 修复当前已识别的功能缺陷
- 优化前后端实现，提高可维护性与安全性

## 问题点

1. 登录会话冲突
   - 后端依赖 `express-session` 存储 `req.session.user`
   - 同一域名浏览器标签页共享 cookie，导致同一浏览器中只能存在一个有效登录会话
   - 前端使用 `sessionStorage.user` 只是客户端缓存，不能替代后端 session 鉴权

2. 私聊页面参数校验不完善
   - `id==NaN` 判断不生效，应使用 `Number.isNaN(id)` 或更严格的校验

3. `/user/isLogin` 接口逻辑错误
   - 使用了 `res.session.user`，应改为 `req.session.user`

4. 安全与部署风险
   - 密码明文存储与传输
   - 输入输出未做 HTML 转义，存在 XSS 风险
   - 前端 AJAX 接口地址硬编码，部署环境切换成本高
   - 在线用户状态仅存在内存，无法支持多实例或重启恢复

## 任务列表

### 1. 会话与测试改造

1.1 支持多会话测试方案

- 调查并决定使用:
  - 不同浏览器/隐身模式测试，或
  - 通过 URL 参数/请求头载入不同 session id，或
  - 改造为基于 token 的鉴权（更推荐）

  1.2 若临时测试需求仅为验证多账号并行

- 在文档中明确“同一浏览器单一域名下，session cookie 共享，建议使用不同浏览器或隐身窗口进行测试”

  1.3 若要实现同一浏览器跨账号并行

- 将后端登录改造为支持 `Authorization: Bearer <token>` 或自定义 `X-Session-Token`
- 前端登录时返回 token，后续请求带 token 进行鉴权
- 后端不再完全依赖默认 cookie session，而是使用 token 绑定用户状态

### 2. 后端修复与优化

2.1 修复 `/user/isLogin` 逻辑

- 用 `req.session.user` 判断登录态

  2.2 修复私聊 ID 校验

- 使用 `Number.isNaN(id)` 或 `if (!id || Number.isNaN(id))` 来判断非法目标用户 id

  2.3 优化在线用户管理

- 考虑将在线状态从内存迁移到持久存储（数据库或 Redis）
- 或增加 session 过期与状态同步机制

  2.4 改进跨域与部署

- 将 AJAX 地址提取为前端配置，避免硬编码 `http://127.0.0.1:5050`
- 考虑使用相对路径或统一 `BASE_URL`

### 3. 安全与数据处理改进

3.1 密码安全

- 使用 `bcrypt` 对注册密码进行哈希后存储
- 登录时比对哈希值

  3.2 输入输出防护

- 后端对用户输入进行严格校验与过滤
- 前端渲染消息时对内容做 HTML 转义
- 防止 XSS 和注入攻击

  3.3 会话安全

- 将 session cookie 设置 `httpOnly`、`sameSite`，根据部署环境启用 `secure`
- 若改造 token 鉴权，避免暴露敏感信息

### 4. 前端改进

4.1 抽离配置

- 在 `frontend/js/base.js` 或单独配置文件中定义后端接口基地址
- 将所有 AJAX URL 改成基于该配置变量

  4.2 统一错误显示

- 在登录/注册/聊天页面提供统一的错误提示区域
- 避免仅依赖控制台调试信息

  4.3 私聊页面优化

- 在私聊页面启动时校验 `id` 是否有效，若无效直接跳转或提示
- 显示“正在与 X 私聊”的清晰上下文信息

## 推荐实施顺序

1. 修复后端现有逻辑错误：`/user/isLogin` 和私聊 ID 校验
2. 明确测试方式：文档说明不同浏览器/隐身窗口进行多账号测试
3. 改造前端配置与接口地址提取
4. 设计并实现 token 鉴权方案（若目标是同一浏览器并行多账号）
5. 扩展在线用户状态存储方案（数据库/Redis）
6. 加入密码哈希与输入防护
